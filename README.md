## 1. Описание

**haibrid** — это легковесный оркестратор для создания отказоустойчивых детерминированных конвейеров обработки данных с использованием ИИ. 

Платформа спроектирована для работы в условиях ограниченных ресурсов и обеспечивает полный контроль над совокупной стоимостью владения (TCO). **Инференс от 2 ГБ RAM**.

Платформа реализует **гибридный инференс**: локальные модели (Sentence-transformers, компактные LLM) выполняют черновую фильтрацию, дедупликацию и классификацию, а в облачные API отправляются только те данные, которые действительно требуют сложной логики. Это позволяет снизить затраты на токены до 90% по сравнению с прямыми вызовами API.

**Важно:** 
- **haibrid** — это оркестратор, а не библиотека AI-моделей. Платформа предоставляет инфраструктуру (статусы, изоляцию, контроль памяти).
- **прикладная логика обработки (какие модели вызывать, когда идти в облако, какие пороги использовать) полностью определяется разработчиком процессора**. 
- Вы сами решаете, как построить гибридный инференс под вашу задачу.

## 2. Для кого

### Ключевые преимущества

- **Экономия (TCO):** Снижение затрат на инфраструктуру и токены за счет приоритетного использования локальных моделей. Вы платите копейки за фильтрацию на своих серверах и только за сложную логику в облаке.
- **Надежность (Reliability):** Отказоустойчивая архитектура. Падение одного «тяжелого» процессора (ML-модели) не останавливает работу всего ядра и других обработчиков. Статусная модель в БД позволяет продолжить обработку с места сбоя даже после перезагрузки системы.
- **Гибкость и безопасность:** Возможность работы в закрытом контуре (Air-gapped) без доступа в интернет. Вся чувствительная первичная обработка данных остается на ваших серверах (on-premise).
- **Масштабируемость:** Горизонтальное масштабирование путем добавления новых процессоров (воркеров) к единой базе данных. Это позволяет обрабатывать миллионы элементов на простых VPS или промышленных ПК.

### Кому подходит

| Аудитория | Что получает |
|-----------|--------------|
| **AI-стартапы** | Быстрый запуск MVP с возможностью бесшовного перехода в production без переписывания логики под требования стабильности |
| **Агрегаторы данных** | Обработка миллионов единиц контента (новости, товары, документы) с минимальными затратами на инфраструктуру |
| **Enterprise** | Создание закрытых on-premise контуров обработки данных с исключением утечки данных в публичные облака |

## 3. Use Cases

| Аудитория | Что получает |
|-----------|--------------|
| **1. Умная пре-модерация** | Локальная модель фильтрует спам и дубли на 2 ГБ RAM. В облако уходит только «сложный» контент для финального вердикта. |
| **2. Дедупликация новостного потока** | Сравнение новых постов с вектором базы через sentence-transformers. Отсечка повторов до этапа дорогого перевода или анализа. |
| **3. Анонимизация персональных данных** | Процессор находит лица, номера машин или паспортов и замазывает их локально. В облачные хранилища уходят уже «безопасные» файлы. |
| **4. Мониторинг репутации (Fintech)** | Сбор упоминаний → определение тональности (локальный BERT) → проверка на ботов → категоризация жалобы → если техсбой → запрос к логам → формирование отчета. |
| **5. Анализ безопасности на производстве (CV)** | Поток кадров с камер → детекция людей и техники (YOLO) → проверка наличия СИЗ → трекинг нарушителя → распознавание лица → формирование инцидента. |
| **6. Автоматизированный скоринг коммерческих предложений (B2B)** | Классификация входящих писем → парсинг номенклатуры (NER) → запрос к ERP → поиск аналогов через эмбеддинги → проверка маржинальности → статус ready_for_manager. |
| **7. Интеллектуальный архив техдокументации** | PDF/TIFF → определение структуры (чертеж/спецификация) → вырезка штампа → OCR → проверка дублей → извлечение текста → семантический поиск связей → при противоречиях → LLM → запись в векторное хранилище. |
| **8. Конвейер медицинской диагностики (МРТ/КТ)** | Получение DICOM → анонимизация метаданных → локальная сегментация органов → поиск аномалий → если аномалия > 2мм → ансамблевая модель → подбор похожих случаев → формирование пре-отчета врачу. |
| **9. Конвейер оцифровки архивов** | Локальный OCR (Tesseract/EasyOCR) распознает текст со сканов на 2 ГБ RAM → статус parsed. Только если качество распознавания низкое (low confidence), элемент уходит на облачную Vision LLM для уточнения. |

 И многое другое, где нужен жестко ограниченый по ресурсам (от 2 ГБ RAM) детерминированный пайплайн с предсказуемым результатом и минимальной стоимостью владения.

## 4. Архитектура и технические детали

### 4.1. Архитектура

- **Configuration-as-Code:** Вся логика пайплайна описывается в YAML-конфигах. Изменение схемы движения данных не требует переписывания ядра.
- **Детерминированный пайплайн**: предсказуемость обработки и гарантированный результат, малое количество галлюцинаций при работе специализированных моделей.
- **Единица обработки: `item`**: Это может быть что угодно: текст, изображения, звук; платформе важно знать только `item_id` и статус.
- **Горизонтальное масштабирование**: Можно запускать любое количество независимых экземпляров процессора (воркера) на разных узлах, работающих с единой базой данныхлюбое.

### 4.2. Статусная модель

Архитектура построена на четких переходах состояний в БД. Каждый элемент движется по цепочке статусов, а процессор может генерировать несколько исходящих веток.

**Пример линейного пайплайна:**
```
raw → filtered → enriched → final
```

**Пример с ветвлением:**
```yaml
processors:
  - name: "deduplicator"
    input_status: "raw"
    success_status: "unique_item"    # ветка 1
    reject_status: "duplicate"        # ветка 2
```

**Возвраты:** достаточно обновить статус элемента в БД на предыдущий — процессор подхватит его снова.

### 4.3. Управление памятью

- **Ленивая загрузка:** модель загружается в память только в момент вызова 
- **Выгрузка весов:** освобождает память после завершения обработки
- **Режимы работы:** при малом объеме RAM (2 ГБ) модели будут перегружаться регулярно, что влияет на скорость обработки. При достаточном объеме памяти модели загрузятся при первом вызове и останутся в кэше

### 4.4. Изоляция и отказоустойчивость

- **Subprocess isolation:** каждый тяжелый процессор (AI-модели, внешние API) работает в изолированном подпроцессе. Утечка памяти или падение процессора (воркера) не влияет на мастер.
- **Watchdog + Heartbeat:** мастер все время проверяет пульс каждого подпроцесса и контролирует его потребление памяти. Предусмотрено и мягкое, и жесткое завершеие подпроцесса.
- **Автоматическое восстановление задач:** если процессор был прерван/остановлен не завершив обработку `item`, то благодаря транзакционной модели он опять возьмет в работу этот `item`. В обработку заложен паттерн poison pill.
- **Crash Protection:** падение одного процессора не задевает другие процессоры и ядро.

### 4.5. Блокировка элементов в БД

Для предотвращения конфликтов при горизонтальном масштабировании используется блокировка записей на уровне Postgres: `SELECT ... FOR UPDATE SKIP LOCKED`, что позволяет запустить один процессор (воркер) сразу на нескольких нодах.

**Планы развития:** автоматический шардинг на уровне ядра с указанием `replicas: N` в конфиге.

### 4.6. Polling vs Event-driven

Платформа использует polling (циклический опрос БД), а не событийную шину. Это осознанный выбор:

- **Простота:** на одну точку отказа меньше (не нужен Redis/RabbitMQ)
- **Персистентность:** статусы хранятся в БД, при падении процессора элемент не теряется
- **Достаточная производительность:** операции с БД ничтожны по времени по сравнению с загрузкой моделей и AI-инференсом

### 4.7. Стек и инфраструктура

| Компонент | Требования |
|-----------|------------|
| **Язык** | Python 3.10+ (asyncio, asyncpg/pgvector, aiosqlite) |
| **База данных** | PostgreSQL (для масштабирования) или SQLite (для edge-решений) |
| **Операционная система** | Linux (Ubuntu) |
| **Минимальная RAM** | 2 ГБ (OS + PostgreSQL + **haibrid**) |

### 4.8. Наблюдаемость (Observability)

Платформа предоставляет следующие данные для мониторинга:

- Количество `items` (элементов) в каждом статусе
- Время обработки батча `items`
- Активные процессоры и их heartbeat
- Решения ResourceManager: запуск/остановка процессоров, количество процессоров по статусам, использование памяти
- Время загрузки моделей

Для визуализации рекомендуется подключать **Grafana** с PostgreSQL-экспортером. Встроенного дашборда платформа не предоставляет.

### 4.9. Версионирование моделей

Модели не меняются на лету. При обновлении `config.yaml` (новая модель, новые параметры) процессор перезапускается. **Новые элементы**, поступающие в пайплайн после рестарта, будут обработаны новой версией модели.

Элементы, которые уже находятся в пайплайне (между статусами), останутся в старой цепочке. Для миграции рекомендуется после рестарта запустить backfill-процессор для переобработки старых элементов

### 4.10. Air-gapped режим

Платформа может работать полностью без доступа в интернет. Для этого:

1. **Первичная загрузка моделей:** выполняется на машине с интернетом, кэш HuggingFace (`~/.cache/huggingface`) копируется на air-gapped сервер
2. **Обновление моделей:** либо копирование кэша с машины stage/test, либо доставка через Docker-образ
3. **Переменные окружения:** `HF_HUB_OFFLINE=1` и `TRANSFORMERS_OFFLINE=1` отключают попытки обращения в сеть

### 4.11. Контроль стоимости облачного инференса

Платформа предоставляет базовые механизмы для контроля затрат:

- **Методы для расчета лимитов:** в репозитории есть функции для подсчета использованных токенов за период
- **Механизм извещения:** при приближении и превышении лимита можно отправить уведомление администратору

На уровне процессора можно подключить **LiteLLM**, который поддерживает лимиты на количество запросов и токенов за период. При достижении лимита процессор может перейти в режим ожидания или вернуть статус `rate_limited` для дальнейшей обработки.

**Пример настройки лимитов в процессоре:**
```python
class RateLimitedProcessor(ProcessorCommonMixin, CyclicScheduler):
    async def _process_single_item(self, item_id):
        # Проверка лимита через LiteLLM или самописный счетчик
        if not await self._check_rate_limit():
            await self.repository.update_status(item_id, self.rate_limit_status)
            await self.notify_admin("Rate limit exceeded")
            return False
        
        # Обычная обработка
        result = await self._call_llm(item)
        await self._increment_token_counter(result.usage)
        await self.repository.update_status(item_id, self.success_status)
        return True
```

**Планы развития:** встроенные quotas на уровне конфигурации процессора (`max_tokens_per_hour`, `max_requests_per_day`) с автоматической приостановкой обработки при превышении.

## 5. Конфигурация и примеры

### 5.1. Пример процессора

**Код процессора** (`src/my_app/processors/deduplicator.py`):
```python
from haibridcore.base import CyclicScheduler
from haibridcore.mixins import ProcessorCommonMixin, AlModelMixin

class LocalDeduplicator(ProcessorCommonMixin, AlModelMixin, CyclicScheduler):
    async def _process_single_item(self, item_id: int) -> bool:
        item = await self.repository.get_item_by_id(item_id)
        model = await self.get_sentence_model()
        
        threshold = self.config.get("similarity_threshold", 0.85)
        embedding = model.encode(item['content'])
        is_duplicate = await self.repository.find_similar(embedding, threshold=threshold)
        
        next_status = self.reject_status if is_duplicate else self.success_status
        await self.repository.update_status(item_id, next_status)
        return True
```

**Конфигурация процессора** (`config.yaml`):
```yaml
processors:
  - name: "deduplicator"
    class: "my_app.processors.deduplicator.LocalDeduplicator"
    input_status: "raw"
    success_status: "unique_item"
    reject_status: "duplicate"
    similarity_threshold: 0.90
    model: "all-MiniLM-L6-v2"
    required_memory_mb: 600
```

### 5.2. Пример цепочки из двух процессоров

**Кейс:** Сначала бесплатно проверяем уникальность локально, и только если данные новые — отправляем их в облачное API.

**Конфигурация пайплайна** (`config.yaml`):
```yaml
processors:
  # Шаг 1: Локальная фильтрация (экономия)
  - name: "deduplicator"
    class: "my_app.processors.deduplicator.LocalDeduplicator"
    input_status: "raw"
    success_status: "unique_item"
    reject_status: "duplicate"
    similarity_threshold: 0.90
    model: "all-MiniLM-L6-v2"
    required_memory_mb: 600

  # Шаг 2: Облачное обогащение (только для уникальных)
  - name: "enricher"
    class: "my_app.processors.openai_enricher.OpenAIEnricher"
    input_status: "unique_item"
    success_status: "ready_to_publish"
    rate_limit_status: "enricher_rate_limit"
    api_key: "{{ SECRET_API_KEY }}"
    required_memory_mb: 50
```

## 6. Интеграции и симбиозы с фреймворками и платформами

### 6.1. **haibrid** + n8n

**Роли:**
- **n8n** — фронтенд: триггеры (IMAP, webhook, расписание), визуальные workflow, интеграции с SaaS (CRM, Slack, Google Sheets)
- **haibrid** — бэкенд: AI-обработка, контроль памяти, гибридный инференс, статусная модель

**Кейс: Email → AI → CRM**
1. n8n захватывает письмо через IMAP-триггер
2. n8n создает новый `item` в **haibrid** приложении (прямой вызов Postgres либо через endpoint)
3. **haibrid**: OCR → классификация → дедупликация → извлечение полей (локально → при необходимости GPT)
4. **haibrid** возвращает структурированный JSON (Webhook), либо Polling со стороны n8n
5. n8n создает сделку в CRM и отправляет уведомление в Slack

**Ценность:** n8n не нужно держать модели AI, **haibrid** не нужно реализовывать интеграции с SaaS. 80% документов обрабатываются локально.

---

### 6.2. **haibrid** + Prefect

**Роли:**
- **Prefect** — мета-оркестратор: координация разных пайплайнов, централизованные расписания, единый дашборд, алерты
- **haibrid** — специализированный AI-оркестратор: управление статусами, памятью, гибридным инференсом внутри своего пайплайна

**Кейс: Data Warehouse + AI обогащение + ML тренинг**
1. Prefect запускает dbt (ETL)
2. После успеха — запускает **haibrid** для AI-обогащения
3. После успеха **haibrid** — запускает ML-тренинг
4. Все ошибки логируются в Prefect UI, алерты в Slack

**Ценность:** **haibrid** не нуждается в Prefect для своего DAG, retry или мониторинга. Prefect добавляется когда **haibrid** — один из многих компонентов в большой инфраструктуре.

---

### 6.3. **haibrid** + LangChain

**Роли:**
- **haibrid** — оркестратор пайплайна: статусная модель, контроль памяти, изоляция, гибридный инференс
- **LangChain** — внутри процессора: сложная агентная логика, динамические инструменты, цепочки рассуждений

**Кейс: Анализ юридических контрактов**
1. **haibrid** управляет пайплайном: `raw_contract → ocr_extracted → analyzed → risky? → lawyer_review / archived`
2. Внутри процессора `analyzed` используется LangChain:
   - Извлечение структуры (PydanticOutputParser)
   - Анализ рисков с Chain of Thought
   - Генерация альтернативных формулировок
3. **haibrid** решает, эскалировать юристу или нет

**Ценность:** **haibrid** управляет долгими паузами (юрист может смотреть неделю), изолирует процессор (падение LangChain не валит всю систему), контролирует память. LangChain дает гибкость для неструктурированных контрактов.

## 7. Сравнение с аналогами

| Критерий | **haibrid** | **n8n / Flowise** | **LangChain / LangGraph** | **Prefect / Dagster** | **CrewAI / OpenCLAW** |
|----------|-------------|-------------------|--------------------------|----------------------|----------------------|
| **Основная задача** | AI-конвейеры с контролем ресурсов | No-code автоматизация | LLM-приложения и агенты | Data-оркестрация | Мультиагентные системы |
| **Метод работы** | Последовательный конвейер |	Визуальный поток |	State-machine граф |	Data-пайплайн |	Мультиагентный хаос |

## 8. Ограничения и особенности

- **Холодный старт:** Первая загрузка весов моделей требует времени (зависит от диска и сети). При малом объеме RAM (2 ГБ) модели будут перегружаться регулярно, что влияет на скорость обработки.
- **Оркестрация инфраструктуры:** Платформа масштабируется добавлением инстансов (нод) с разной конфигурацией процессоров, но управление парком этих инстансов (оркестрация контейнеров) ложится на внешние инструменты (Docker Compose, Kubernetes).
- **Специфичные загрузчики:** Базовое ядро включает стандартные загрузчики HuggingFace. Специфические методы загрузки или проприетарные API реализуются на уровне прикладных процессоров.
- **GPU:** Управление памятью GPU находится в разработке. На текущий момент GPU поддерживается через параметры `device` и `dtype`, но контроль VRAM не автоматизирован. Рекомендуется контейнеризация для изоляции GPU-воркеров.
- **Горизонтальное масштабирование:** Реализовано через запуск независимых процессоров (воркеров) на разных физических нодах. Состав процессоров на конкретном узле определяется его конфигурацией, при этом каждый тип процессора запускается на ноде в единственном экземпляре.

## 9. Планы развития

Платформа активно развивается. В ближайших планах:

| Направление | Планируемые возможности |
|-------------|------------------------|
| **Масштабирование** | Автоматический шардинг процессоров с указанием `replicas: N` в конфиге. |
| **GPU** | Интеграция с `nvidia-smi` для контроля VRAM. Поддержка FIFO-очереди для GPU-памяти аналогично CPU. Возможность указания `device: "cuda:0"` с автоматическим распределением моделей между GPU. |
| **Контроль стоимости** | Встроенные quotas на уровне конфигурации процессора: `max_tokens_per_hour`, `max_requests_per_day`. Автоматическая приостановка обработки при превышении лимита с уведомлением администратора. |
| **Наблюдаемость** | Встроенный дашборд для мониторинга очередей и статусов в реальном времени. Экспорт метрик в Prometheus. |

## 10. Поставка и внедрение

Платформа поставляется как скомпилированный пакет. Для запуска production-конвейера необходимо 

1. **Прикладная часть**:
- **База данных** - описать схему прикладной части, поля `item`, дополнительные требуемые таблицы  (полная схема включает также таблицы для эмбеддингов, heartbeat, команд и аудита)
- **Источник данных** - процессор поставляющий в пайплайн `item` со статусом `raw`: веб-скраппер, ендпоинт, отдельная поставка данных напрямую в БД
- **Процессоры (воркеры)** - написать код для прикладной обработки `item` в пайплайне. Для примера приведен [промпт](#11-vibe-code-prompt).
- **Настроить config.processors.yaml** — описать процессоры, статусы, модели, пороги, параметры для ResourceManager
2. **Оценочный запуск** - для каждого процессора в изолированом режиме провести запуск и  подтвердить лимиты памяти для ResourceManager.
3. **Запустить мастер** — `python -m haibrid` как сервис.
5. **(Опционально) Масштабировать** — запустить дополнительные инстансы процессоров (воркеров) на других нодах, подключив их к той же БД
